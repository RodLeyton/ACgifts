Importing neighbors via CSV file

A template file for the csv format can be found here. It is also in the application directory (you can find that with the Dir button on the edit form).
Please refer to this file in a text editor if you're having import issues, it has many examples of acceptable formats including special characters and dates.

The easiest way to edit csv files is in a spreadsheet application such as MS Excell or LibreOffice/Openoffice Calc.
If any errors occur you will recieve a message directing you to check the error log

Fields in the csv file

Name

This is the forum or acual name of your neighbor, currently it is only displayed as a tooltip when you hover over the game names, in the future I may add a details panel

Group

Either a fuel ring name, or something like main. If blank will be grouped under 'unassigned'.

NameSend / NameRecv

The game name of the account you send to or receive from. Obviously these are different in fuel rings, but they may be the same on your main account.

CodeSend / CodeRecv

At present these are not displayed, and may be left empty. However in future an extra details form may be added.

LastSend / LastRecv

The date and time of the last gift sent and recieved. This may be left blank, or be a valid format

CntSend / CntRecv

The count of gifts sent and recieved since this neighbor was added. If left blank defaults to 0.
This value is used to calculate the gifting rate (or reliability) and should reflect to the number of gifts since the date in the 'Added' field.
In other words, if you specify the neighbor was added 10 days ago (and they gift every day) the count shoud be 10.

Added

The date and time this neighbor was added to your list. This may be left blank, or be a valid format
If left blank it will default to the current date. This value to used to calculate the gifting rate (or reliability).

Valid date time formats

All standard formats are accepted, some of which depend on your locale, along with many non-standard variations.
Take note any locale dependent formats differ depending on what location your computer is set to. This is by design and usually is a very helpfull convenience.
For example the short date format '01/02/2023' will refer to Janurary in the US and Feburary in France. If you stick to what is normally used in your country (and your computers locale is correct) you should be fine.
If any date time that fails to parse correctly during an import you will be notified, and the logfile will contain the specific line number and contents that caused the issue.
The log file can be found here
It is worth pointing out that when using a spreadsheet if you set the cell format to a datetime type (instead of text) you should avoid any hassle as it will format it correctly for your locale,
which means it will be parsed correctly by ACgifts based on your locale.

Some examples follow

Short date (locale dependant)
  • US - MM/dd/yyyy
  • France - dd/MM/yyyy
  • Japan - yyyy/MM/dd
  • Denmark - dd.MM.yyyy
General date time (locale dependant)
  • Short date (as above) with time e.g. in the US   'MM/dd/yyyy HH:mm'
ISO 8601 (universal standard)
  • yyyy-MM-ddThh:mm:ss   e.g. 2023-01-20T09:30:00 => 20 Jan 2023 9:30 am
Non-standard formats

Most unambigous natural language formats will also work, such as;

  • 'April 10, 2023' - works as expected in any locale.
  • '1 march 2023 9pm' - works as expected in any locale.
  • '1 mar 2023 6am' - works as expected in any locale.
  • '1st march 2023 9pm' - fails to parse! (Don't use st,nd,rd,th after the day)
  • 'half past a sparrows fart' - I believe this only works in the Australian outback.

Where can I find the logfile?

Logs are located in your local application directory, under CrashnBurn subfolder.
Usually this is 'C:\Users\[name]\AppData\Local\CrashnBurn\ACgifts_Release\log-release.txt'
The exact location will depend on your OS, but the easiest way to find it is to click the Dir or Log buttons on the Edit form.
When the primary log reaches 250kB it is moved to 'log-release.bak.txt' and a new file is started.

If you have any issues, please copy this file at the time and provide a description of what you were doing when asking for help.

Basically I'm too cheap to pay for a certificate. But then again would you pay for one?
This is a not for profit project, and I don't mean like all thouse other non-profits that take in millions per year.
Lets not forget you have the ability to compile the source code yourself without a single warning.
Even then a security certificate only verifies identity, not the code signed by it.

Lets look at the warnings...

Windows protected your PC, Unrecognised app prevented from starting

The warning is "I haven't seen this app before"
Considering I just made it, it is to be expected.
To continue click 'More info' and select 'Run anyway'.
But before you do, have a quick read here as to why this warning comes up and the 2 teirs of speed and cost to remove it.

The real issue is everytime an update is released any recognition from SmartScreen is lost, unless thousands of people are installing the app it is unlikely this warning will go away.

To verify a publisher a code signing certificate must be used, not exactly a cheap proposition considering this is a free app from a private person.

Why are there so many .dat files?

Safety of data is the top priority! Every time the app is started a backup of the previous sessions data is saved.
These are rotated so the largest number (max 30) is the oldest.
Normally this should cover the last month, so we have plenty of options to recover data if anything goes wrong!
In addition, before a restore or import a backup is automatically created named 'ACgifts_backup_pre...' since this occurs unfrequently these are never deleted.
The main data file is 'neighbors.dat' everything else can be deleted (even 'app.config')
