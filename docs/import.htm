Importing neighbors via CSV file

A template file for the csv format can be found here. It is also in the application directory (you can find that with the Dir button on the edit form).
Please refer to this file, it has many examples of acceptable formats.

The easiest way to edit csv files is in a spreadsheet application such as MS Excell or LibreOffice/Openoffice Calc.
If any errors occur you will recieve a message directing you to check the error log

Fields in the csv file

Name

This is the forum or acual name of your neighbor, currently it is only displayed as a tooltip when you hover over the game names, in the future I may add a details panel

Group

Either a fuel ring name, or something like main. If blank will be grouped under 'unassigned'.

NameSend / NameRecv

The game name of the account you send to or receive from. Obviously these are different in fuel rings, but they may be the same on your main account.

CodeSend / CodeRecv

At present these are not displayed, and may be left empty. However in future an extra details form may be added.

LastSend / LastRecv

The date and time of the last gift sent and recieved. This may be left blank, or be a valid format

CntSend / CntRecv

The count of gifts sent and recieved since this neighbor was added. If left blank defaults to 0.
This value is used to calculate the gifting rate (or reliability) and should reflect to the number of gifts since the date in the 'Added' field.
In other words, if you specify the neighbor was added 10 days ago (and they gift every day) the count shoud be 10.

Added

The date and time this neighbor was added to your list. This may be left blank, or be a valid format
If left blank it will default to the current date. This value to used to calculate the gifting rate (or reliability).

 

Valid date time formats

All standard formats are accepted, some of which depend on your locale.
Take note any locale dependent formats differ depending on what location your computer is set to. This is by design and usually is a very helpfull convenience.
For example the short date format 01/02/2023 will refer to Janurary in the US and Feburary in France. If you stick to what is normally used in your country (and your computers location is correct) you should be fine.
It is worth pointing out that when using a spreadsheet if you set the cell format to a datetime type (instead of text) you should avoid any hassle as it will format it correctly for your locale.

Some examples follow

Short date (locale dependant)
  • US - MM/dd/yyyy
  • France - dd/MM/yyyy
  • Japan - yyyy/MM/dd
  • Denmark - dd.MM.yyyy
General date time (locale dependant)
  • Short date (as above) with time e.g. in the US   'MM/dd/yyyy HH:mm'
ISO 8601 (universal standard)
  • yyyy-MM-ddThh:mm:ss   e.g. 2023-01-20T09:30:00 => 20 Jan 2023 9:30 am
Non-standard formats
  • Most normal language formats will also work so long as it is unambigous
  • 'April 10, 2023' - works as expected in any locale.
  • '1 march 2023 9pm' - works as expected in any locale.
  • '1 mar 2023 6am' - works as expected in any locale.
  • '1st march 2023 9pm' - fails to parse!
  • 'half past a sparrows fart' - only works in the Australian outback.
